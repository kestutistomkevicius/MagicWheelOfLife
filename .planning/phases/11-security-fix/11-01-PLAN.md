  <files>supabase/migrations/20260323000001_profiles_tier_column_security.sql</files>
  <action>
Create the migration file with exactly these three SQL statements:

```sql
-- Revoke the implicit table-level UPDATE grant from authenticated.
-- After this, any UPDATE that touches `tier` from browser Supabase client fails with
-- "permission denied for column tier" at the PostgreSQL level.
REVOKE UPDATE ON TABLE public.profiles FROM authenticated;

-- Re-grant UPDATE only on columns users are legitimately allowed to change.
GRANT UPDATE (avatar_url, color_scheme) ON TABLE public.profiles TO authenticated;

-- Defensive GRANT SELECT — Supabase/PostgREST may reject wildcard selects on tables
-- where column-level grant is active. This is idempotent and clarifies intent.
GRANT SELECT ON TABLE public.profiles TO authenticated;
```

The existing `profiles: update own` RLS policy must NOT be touched — it stays and continues to
enforce row-level ownership (user.id = auth.uid()). Column-level grant enforces which columns
are writable within those rows.

After writing the file, apply it: `npx supabase db reset`
(Full reset is required — see RESEARCH.md Pitfall 2. Do not use `migration up` only.)

Verify the fix using psql immediately after reset:

```sql
-- Must return: ERROR: permission denied for column tier
```

Also verify SELECT still works:

  <action>
Create `supabase/functions/set-tier/index.ts`. Pattern from Phase 9 `ai-coach` function (same project).

```typescript
import 'jsr:@supabase/functions-js/edge-runtime.d.ts'
import { createClient } from 'npm:@supabase/supabase-js@2'

const corsHeaders = {
  'Access-Control-Allow-Origin': '*',
  'Access-Control-Allow-Headers': 'authorization, content-type',
}

Deno.serve(async (req) => {
  if (req.method === 'OPTIONS') {
    return new Response('ok', { headers: corsHeaders })
  }

  const authHeader = req.headers.get('Authorization')
  if (!authHeader) {
    return new Response('Unauthorized', { status: 401, headers: corsHeaders })
  }

  // Validate the calling user's JWT
  const userClient = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_ANON_KEY')!,
    { global: { headers: { Authorization: authHeader } } }
  )
  const { data: { user }, error: authError } = await userClient.auth.getUser()
  if (authError || !user) {
    return new Response('Unauthorized', { status: 401, headers: corsHeaders })
  }

  // Validate request body
  let tier: 'free' | 'premium'
  try {
    const body = await req.json() as { tier?: unknown }
    if (body.tier !== 'free' && body.tier !== 'premium') {
      return new Response('Invalid tier', { status: 400, headers: corsHeaders })
    }
    tier = body.tier
  } catch {
    return new Response('Invalid JSON', { status: 400, headers: corsHeaders })
  }

  // Use service-role client — bypasses RLS, the only path allowed to write tier
  const serviceClient = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  )
  const { error } = await serviceClient
    .from('profiles')
    .update({ tier })
    .eq('id', user.id)

  if (error) {
    return new Response(error.message, { status: 500, headers: corsHeaders })
  }

  return new Response(JSON.stringify({ ok: true }), {
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
})
```

**Do NOT** add any dev/prod environment guard to the Edge Function itself. The SettingsPage UI
already guards the tier toggle with `import.meta.env.DEV`. The function must work identically
in all environments — it is the legitimate server-side tier write path everywhere.

**Do NOT** use the service-role key in any frontend code. It must only appear in
`Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')` inside this Edge Function.
